## §2-3. Train/holdout split — реальная картина

Глава §3.1 требует «железную стену» между train и test, и чтобы holdout был **позже по времени**, не случайной выборкой строк.

Проверил [evaluation.py:26-57](../../src/detect_temperature/evaluation.py#L26-L57) — функция `time_ordered_split`:

```python
dates = sorted(str(value) for value in frame[date_column].dropna().unique() ...)
split_idx = int(len(dates) * (1 - test_fraction))
train_dates = set(dates[:split_idx])  # первые 67%
test_dates  = set(dates[split_idx:])  # последние 33%
```

Это **честный** time-ordered split, не случайный. Глава 5 этого опасалась — нет повода. Это важный плюс проекта.

## §4. Look-ahead bias — реальная угроза

Глава §4.2 перечисляет 4 пути утечки. Что в проекте:

| Канал утечки | Риск в нашем коде |
|---|---|
| (a) Перерасчёт прогнозов задним числом | **Реальный риск.** Open-Meteo Historical Forecast API не публикует, был ли это live прогноз на тот день, или reanalysis. Это нельзя проверить из API. |
| (b) Статистика на всём периоде | Нет. Признаки [features.py](../../src/detect_temperature/features.py) — только лёгкие per-row (`station_lat`, `day_of_year_sin`, etc.) |
| (c) Сдвиг резолва | Нет. `target_extreme=max` берёт max за `target_date`, прогноз — за тот же день. Не вижу пересечения |
| (d) Утечка через ID | **Возможна.** `station_id_hash` в признаках. GBM может выучить «это station_id_hash=4663 → bias −0.3 °C» — это уже не модель, это lookup table |

Канал (a) — фундаментальный, нельзя починить. Канал (d) — теоретический, но мы только что увидели его эффект эмпирически: на in-time holdout MAE = 0.455 °C, на OOT 30-дневный holdout (свежие 30 дней) MAE = 0.473 °C. Разница маленькая (0.018 °C), значит overfitting не катастрофичен. Это лучшая новость дня.

## §6. Калибровка ≠ точность

**Главная находка этой главы для нашего проекта.** Глава 2 предсказывала «модель недооценивает хвосты». Когда я измерил reliability через PIT histogram, картина оказалась сложнее.

PIT (probability integral transform): для каждого OOT-наблюдения считаем `Φ((observed - predicted) / σ_station)`. Если σ честная, эти числа равномерны в [0, 1].

Что получилось на 3060 OOT-точек (последние 30 дней):

```
bin    count   expected  deviation
0.0-0.1   44       306    -86%   ← очень мало
0.1-0.2   87       306    -72%
0.2-0.3  140       306    -54%
0.3-0.4  188       306    -39%
0.4-0.5  826       306   +170%   ← пик в центре
0.5-0.6 1092       306   +257%   ← гигантский пик
0.6-0.7  232       306    -24%
0.7-0.8  244       306    -20%
0.8-0.9  130       306    -58%
0.9-1.0   77       306    -75%   ← очень мало
```

Это **U-shape наоборот**. Огромный пик в центре, провал на краях. Что это значит:

> **σ, которую мы используем (`1.5 × per-station MAE`, default `2.5 °C`), слишком ШИРОКА для центра распределения.**

То есть: модель `predicted_c` сама по себе очень точная, гораздо точнее чем sigma говорит. На большинстве дней мы могли бы поставить sigma `0.6-0.8 °C`, и calibration был бы лучше.

**Но!** Tail coverage показывает другую сторону:

```
|z|>k    observed   gauss      ratio
>1        7.58%    31.73%     0.2x   ← мы обещаем больше неуверенности
>2        0.98%     4.55%     0.2x
>3        0.26%     0.27%     1.0x   ← здесь Gaussian угадывает
>4        0.07%     0.01%    10.3x   ← здесь модель не догоняет
```

Дальние хвосты `|z|>4` — реально в 10× чаще, чем Gaussian предсказывает (это глава 2 говорила правильно). А ближние хвосты `|z|<3` — гораздо реже (это глава 2 не предсказывала).

**Вывод:** наш `Normal CDF + σ = 1.5×MAE` плохо подходит для **обеих** сторон распределения. Решение, которое глава 2 §10 уже подсказывала — Student-t или mixture-of-Gaussians. Они умеют одновременно «острый центр + толстые хвосты».

Это меняет приоритет фиксов. Раньше я думал: «sigma слишком узкая, надо расширять». Реально: «sigma подобрана грубо, и для центра она слишком широкая, для хвостов слишком узкая».

## Главный вывод этой главы для проекта

Я ожидал найти overfitting (большой разрыв in-time MAE vs OOT MAE). Нашёл **не его**. In-time 0.455, OOT 0.473 — почти нет разрыва. GBM **здоров**.

Что я **нашёл** — это **mis-calibration sigma**. PIT histogram показал, что наш `σ = 1.5×MAE` плохой prior для нормального распределения. Реальное распределение нормальным просто не является — оно более «острое в центре + толще на хвостах», как Student-t с малыми степенями свободы.

Это конкретный, измеряемый, фиксируемый дефект. Решение — заменить `normal_interval_probability` на `student_t_interval_probability` (или mixture). Но это правка глав 6-7, не главы 5.
